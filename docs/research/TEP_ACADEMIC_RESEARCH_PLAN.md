# TEP Academic Research Plan

This document defines research tracks for the Trust Evidence Protocol.

The goal is not to turn the agent into a rigid theorem prover. The goal is to
preserve heuristic reasoning, abstraction, critique, and hypothesis generation
while forcing commitments to reconcile with source-backed facts.

Core thesis:

```text
Maximize agent freedom during exploration.
Require fact reconciliation before commitment.
Keep every mechanical backend as a candidate producer, not a proof authority.
```

For the full academic reasoning synthesis behind this plan, see
`docs/research/TEP_REASONING_RESEARCH.md`.
For bounded planning, deferral, and curiosity research, see
`docs/research/TEP_PLANNING_CURIOSITY_RESEARCH.md`.

## Research Digest

The research-backed direction is:

- use abduction to generate multiple plausible explanations
- use explanatory coherence to compare how well each explanation fits known facts
- use defeasible argumentation to track support, contradiction, and defeaters
- use truth maintenance and belief revision to update dependent theories when
  assumptions fail
- use formal methods only as scoped slices over explicit claims and constraints
- use LLM self-verification only when anchored to external tools, records, tests,
  or source quotes
- treat large tasks as bounded-rational planning problems where explicit deferral
  is better than superficial coverage
- use value-of-information and active-learning ideas to decide which hypotheses
  should be checked, asked about, or deferred

The resulting TEP behavior should be:

```text
fact-compatible candidate narratives are allowed during exploration
hypothesis chains are valid when marked and consistent
only reconciled fact-backed subsets may justify commitment
```

## Commitment Boundary

TEP should distinguish four reasoning zones.

### Exploration

The agent may freely:

- form hypotheses
- build abstractions
- infer likely causes
- compare with prior patterns
- criticize the current design
- propose alternatives
- use local hypothesis-on-hypothesis chains

Exploration output is useful, but it is not proof.

### Reconciliation

Before the agent uses its view for a plan, edit, permission request, model update,
flow update, claim promotion, or final answer, it must reconcile the view with
canonical records.

The reconciliation pass should classify each important element as:

- supported by active `CLM-*`
- contradicted by active `CLM-*`
- supported only by fallback/historical `CLM-*`
- hypothesis-only
- generated by analogy or pattern recognition
- missing support
- blocked by open questions

### Commitment

Only the reconciled subset may justify action.

Unreconciled parts may remain as:

- tentative `CLM-*`
- `PRP-*` proposal assumptions
- `OPEN-*` questions
- `WCTX-*` local context
- safe exploration probes

They must not become decisive proof.

### Retrospective

After a significant action or failed hypothesis, the agent should be able to
inspect what it believed, which facts supported it, which assumptions failed, and
what records must be updated or demoted.

This is where truth-maintenance and belief-revision research is directly useful.

## Track 1: Heuristic Reasoning And Reconciliation

### Research Question

How can TEP let the agent reason creatively without allowing unsupported
interpretations to become hallucinated commitments?

### Relevant Work

Peircean abduction is useful because it treats hypothesis generation as a
legitimate reasoning activity, not a defect. The important distinction for TEP is
between "worth pursuing" and "proved enough to act on".

Chain-of-Verification is relevant because it separates drafting a view from
planning verification questions and producing a verified answer.

SelfCheckGPT is relevant because it operationalizes a consistency signal:
hallucinated facts tend to diverge across stochastic samples, while better-known
facts tend to be more stable. TEP should not adopt this as proof, but it is a good
candidate signal for "needs reconciliation".

Truth Maintenance Systems are relevant because they record reasons for beliefs
and support dependency-directed revision when assumptions are contradicted.

### TEP Interpretation

TEP should not forbid heuristic reasoning. It should introduce a commitment
boundary:

```text
view -> reconciliation -> commit-supported subset
```

The agent's "view" is allowed to be richer than the facts. The committed output
must say which parts survived reconciliation.

### MVP Experiment

Add a `reconcile-view` workflow before implementing a command.

Input:

- short free-form agent view
- optional record refs
- task intent: answer, plan, edit, permission, model, flow, review

Output:

- supported elements
- contradicted elements
- fallback-only elements
- hypothesis-only elements
- missing facts
- recommended persistence targets
- safe commitment subset

Later command:

```bash
context_cli.py reconcile-view --task "..." --view-file view.md --format text|json
```

### Failure Modes

- The agent may overfit facts to its preferred story.
- The reconciliation output may become boilerplate.
- The agent may mark too much as "missing" and stop being useful.
- The command may become an LLM task disguised as a deterministic tool.

Guardrail: the mechanical command should validate refs, lifecycle, quote support,
and contradictions. It should not judge the semantic quality of the whole view
without canonical support.

## Track 2: Corroboration Over Source Authority

### Research Question

How should TEP handle a trusted source that conflicts with many independent
source-backed facts?

### Relevant Work

Truth-discovery research treats conflicting sources as a core data-integration
problem. Bayesian truth-finding models infer both likely truth and source quality
from conflicting records.

Dempster-Shafer evidence theory is relevant because it addresses evidence
combination under uncertainty and explicitly exposes conflict as a first-class
problem. It also has well-known pitfalls when evidence is highly conflicting or
source independence assumptions are wrong.

### TEP Interpretation

TEP should separate:

- `control authority`: who may authorize or stop actions
- `epistemic support`: what the facts support
- `source reliability`: prior and historical source quality
- `independence`: whether sources really corroborate or share one origin
- `defeaters`: facts that weaken a source or interpretation

User statements have high control authority. They should not have absolute
epistemic authority.

If a user claim conflicts with multiple independent accepted runtime/code claims,
the agent should respectfully challenge the claim and show the conflict:

```text
The user instruction controls what I may do, but the factual claim appears weak:
CLM-A and CLM-B from independent runtime/code sources contradict it.
```

### MVP Experiment

Extend conflict review reports with a non-mutating "corroboration packet":

- claim under review
- supporting claims grouped by independence group
- contradicting claims grouped by independence group
- source kinds involved
- source critique status
- fallback/historical exclusions
- suggested challenge text for user-facing output

Do not calculate a single global truth score at first. Prefer explainable packets.

### Failure Modes

- Majority can be wrong when sources are dependent.
- User may be describing future intent rather than current fact.
- Runtime can be stale, flaky, or environment-specific.
- Source independence may be overestimated.

Guardrail: challenge user claims only when contradictions are source-backed and
the user-facing output preserves user control.

## Track 3: Partial Formal Models Of Software Systems

### Research Question

Can TEP build useful formal models of large software systems without pretending
to fully formalize the entire system?

### Relevant Work

TLA+ shows a practical tradition of modeling programs and systems, especially
concurrent and distributed systems, with simple mathematics.

Model checking gives a way to verify whether an abstract state model satisfies a
formal specification and can produce counterexample executions when it does not.

Lightweight formal methods argue for pragmatic, partial formalization instead of
high-ceremony complete specification.

Alloy is a useful reference for small relational models and bounded analysis.

Z3/SMT is useful for local satisfiability and consistency snapshots over typed
logic projections.

### TEP Interpretation

TEP should not try to create one complete mathematical model of a large system.

It should create **formal slices**:

- scoped entities and symbols
- invariants
- preconditions
- postconditions
- state transitions
- flow oracles
- safety properties
- contradiction candidates
- rollback/impact dependencies

Formal slices should be linked to `CLM-*`, `MODEL-*`, and `FLOW-*`, and should
stay auditable from source quotes.

### MVP Experiment

Define a `formal_slice` schema inside or near `MODEL-*` / `FLOW-*`:

- scope
- symbol refs
- claim refs
- predicates
- invariants
- preconditions
- transitions
- oracle
- solver backend
- generated artifacts

Start with export-only:

```bash
context_cli.py logic-check --solver structural|z3 --closure direct|rules|system
context_cli.py export-formal-slice --model MODEL-* --format smtlib|json
```

### Failure Modes

- The agent may introduce meaningless symbols.
- Formal projection may drift from prose claims.
- The system may be too dynamic for a useful static slice.
- The solver may find artifacts of bad modeling instead of real contradictions.

Guardrail: every formal symbol and atom must trace back to source-backed claims,
and solver output remains a candidate until canonical records are reviewed.

## Track 4: Reasoning, Argumentation, And Defeasible Proof

### Research Question

How can TEP represent reasons, counterarguments, defeaters, and belief updates
without exposing hidden chain-of-thought or forcing all reasoning into predicate
logic?

### Relevant Work

Dung-style abstract argumentation is relevant because it models arguments and
attack relations without requiring one fixed internal structure for every
argument.

Defeasible and non-monotonic reasoning are relevant because software
investigation often produces conclusions that are reasonable until new evidence
defeats them.

AGM belief revision is relevant because TEP needs controlled belief updates when
new facts contradict old models.

Truth-maintenance systems are relevant because they track reasons and
dependencies for beliefs, enabling explanation and revision.

Toulmin/Walton-style argumentation is relevant because it separates claims,
grounds, warrants, backing, qualifiers, rebuttals, and critical questions. That
maps well to user-facing evidence chains without requiring hidden chain-of-thought.

### TEP Interpretation

TEP should model public reasoning artifacts, not hidden chain-of-thought.

Candidate public artifacts:

- evidence chain
- reasoning checkpoint
- reconciliation packet
- defeater list
- argument/counterargument packet
- rollback report
- proposal with risks and stop conditions

### MVP Experiment

Extend `build-reasoning-case` to include:

- supporting claims
- contradicting claims
- known defeaters
- fallback-only dependencies
- unresolved open questions
- proposal refs
- suggested safe conclusion

Then add a test fixture where:

- a high-authority user claim conflicts with multiple independent runtime/code
  claims
- the expected behavior is respectful challenge, not automatic user-trust

### Failure Modes

- Argument packets can become verbose.
- Defeater lists can become bureaucratic.
- The agent may treat argument acceptability as mathematical truth.

Guardrail: argumentation output is decision support. Proof still resolves to
`CLM-*` plus `SRC-*`.

## Track 5: ML And Statistical Candidate Signals

### Research Question

Which statistical tools can reduce agent search/reasoning cost without becoming
unreviewed truth machinery?

### Relevant Work

NMF/LDA topic extraction provides interpretable topic groupings over text.

Hybrid lexical/semantic retrieval can help find related records when exact
keyword search is too narrow or too broad.

NLI and hallucination detectors can produce contradiction or factuality
candidates, but they must remain prefilters.

### TEP Interpretation

ML/statistical tools should be used for:

- candidate grouping
- duplicate-like detection
- contradiction prefiltering
- cleanup prioritization
- guideline applicability hints
- stale-record surfacing

They should not be used for:

- claim status changes
- proof
- permission grants
- automatic source acceptance

### MVP Experiment

Add optional candidate reports:

- `topic-conflict-candidates --backend lexical|nmf`
- `cleanup-candidates --ranker heuristic|topic`
- `record-neighborhood --method keyword|topic|hybrid`

All outputs should include method metadata and record fingerprints.

### Failure Modes

- Topic models can group by vocabulary rather than semantics.
- Embeddings can blur exact technical distinctions.
- NLI can mishandle temporal scope, negation, or code-specific claims.
- Ranking can hide rare but decisive facts.

Guardrail: candidate outputs must default to compact lists with explicit
"navigation only" labels.

## Proposed Implementation Roadmap

1. Document the reconciliation principle in the skill and developer reference.
2. Add deterministic tests for evidence-chain validation around hypothesis-only
   and fallback-only support.
3. Prototype `reconcile-view` as a workflow document before adding a command.
4. Extend conflict reports with corroboration packets grouped by source
   independence.
5. Add formal-slice design notes before expanding Z3 usage.
6. Add one live-agent Docker test where the agent must challenge a trusted but
   contradicted user claim.
7. Only then implement optional ML/NMF or deeper formal backends.

## Design Principle For TEP

The agent should be allowed to think beyond the facts during exploration.

The plugin should become strict only when the agent crosses a commitment
boundary:

- final answer
- plan
- code edit
- permission request
- strictness change
- claim promotion
- model/flow update
- task completion

At that point the agent must show what is fact-backed, what is contradicted, what
is hypothesis-only, and what remains unresolved.
